# Power Load Forecasting 运维建议

## 1. 总体目标

这套系统的核心目标是：

- 稳定保存**历史实际天气**
- 稳定保存**天气预报快照**
- 支持后续负荷预测、电价预测的数据持续积累
- 避免并发冲突、重复抓取、数据缺口和调度失控

当前推荐的生产运行模式是：

- 首次部署时先做**低频全局历史回填**
- 日常运行只保留一个定时任务执行 `sync`
- 如需对局部时间段补密，再单独执行**高频窗口回补**
- 如需再次回填历史数据，先暂停定时同步

## 2. 推荐目录结构

建议 Linux 服务器使用固定目录，例如：

```text
/opt/power_load_forecasting
  ├─ power_load_forecasting/
  ├─ config/
  ├─ requirements.txt
  ├─ README.md
  └─ .venv/

/data/power_load_forecasting
  ├─ raw/
  ├─ curated/
  └─ meta/
```

建议：

- 代码放 `/opt/power_load_forecasting`
- 数据放独立目录或独立磁盘，如 `/data/power_load_forecasting`
- 配置文件中把 `data_root` 改成绝对路径，避免相对路径误写

示例：

```toml
data_root = "/data/power_load_forecasting"
```

## 3. Python 与环境建议

### 推荐版本

建议使用：

- Python `3.11` 或 `3.12`

虽然当前代码已兼容 Python `3.10`，但生产上更推荐 `3.11+`，原因是：

- 原生支持 `tomllib`
- 性能更好
- 后续依赖兼容性更稳

### 虚拟环境建议

统一使用项目内 `.venv`：

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

不要把依赖装到系统 Python。

## 4. 初次部署建议

初次部署建议严格按顺序执行。

### 4.1 拉取代码

```bash
git clone https://github.com/zskfree/Power_Load_Forecasting.git /opt/power_load_forecasting
cd /opt/power_load_forecasting
```

### 4.2 创建环境

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

### 4.3 修改配置

重点检查：

- `data_root`
- `timezone`
- `forecast_days`
- `forecast_snapshot_backfill_default_start_date`
- `forecast_snapshot_backfill_interval_hours`
- `forecast_snapshot_window_interval_hours`
- `[[regions]]`

### 4.4 先验证命令

```bash
python -m power_load_forecasting status
```

### 4.5 执行历史回填

先做低频全局历史预报快照回填：

```bash
python -m power_load_forecasting backfill-forecast-snapshots
```

再回填历史实际天气：

```bash
python -m power_load_forecasting backfill-actual --start-date 2023-01-01
```

如果后续只想对少量关键时间段补密，再单独执行高频窗口回补：

```bash
python -m power_load_forecasting backfill-forecast-window --start-date 2026-03-01 --end-date 2026-04-20
```

### 4.6 手动执行一次增量同步

```bash
python -m power_load_forecasting sync
```

确认正常后再配置定时任务。

## 5. 日常运行建议

## 5.1 日常只跑 `sync`

日常生产运行建议只定时执行：

```bash
python -m power_load_forecasting sync
```

它会：

- 抓取最新 forecast
- 如果 forecast 没变化，则跳过落盘
- 自动补齐历史实际天气到昨天

## 5.2 推荐调度频率

### 方案 A：每小时一次

适合：

- 希望尽量细粒度保留预报变化
- 接受更频繁的 API 请求

优点：

- 预报快照最完整
- 即使模型更新不频繁，也能捕捉变化点

缺点：

- 调度更频繁
- 请求量更大

### 方案 B：每天 4 次

建议时点：

- `00:00`
- `06:00`
- `12:00`
- `18:00`

适合：

- 常规负荷预测
- 资源更节省

### 当前建议

如果服务器资源和网络没问题，建议：

- **生产默认每小时一次**
- 因为 `sync` 已支持 forecast 内容去重，不会机械产生大量重复快照

补充说明：

- 上面的频率建议只针对 `sync`
- 不建议把 `backfill-forecast-window` 做成长期定时任务
- `backfill-forecast-snapshots` 更适合作为一次性初始化或低频维护任务

## 5.3 历史回补双模式建议

当前推荐把历史 forecast 回补拆成两类：

### 低频全局回补

用途：

- 初始化 2026 年之后的历史 forecast 样本
- 控制总请求量，降低触发 Open-Meteo 限流的概率

默认行为：

- 默认起点读取 `forecast_snapshot_backfill_default_start_date`
- 当前默认起点为 `2026-01-01`
- 默认间隔读取 `forecast_snapshot_backfill_interval_hours`
- 当前默认间隔为 `168` 小时，也就是 7 天一个 run

命令：

```bash
python -m power_load_forecasting backfill-forecast-snapshots
```

### 高频局部窗口回补

用途：

- 只对某段关键日期补密
- 不扩大整段历史的请求量

默认行为：

- 默认间隔读取 `forecast_snapshot_window_interval_hours`
- 当前默认间隔为 `6` 小时

命令：

```bash
python -m power_load_forecasting backfill-forecast-window --start-date 2026-03-01 --end-date 2026-04-20
```

推荐原则：

- 平时优先用低频全局回补
- 只有确实需要补密某个时间窗口时，才用高频窗口回补
- 不要把多年历史直接按高频窗口全量扫一遍

## 6. systemd 定时任务建议

## 6.1 推荐使用 systemd timer

优点：

- 比 `cron` 更稳定
- 日志和状态更容易排查
- 支持 `Persistent=true`，机器重启后可补触发

## 6.2 service 示例

```ini
[Unit]
Description=Power Load Forecasting Sync
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
WorkingDirectory=/opt/power_load_forecasting
ExecStart=/opt/power_load_forecasting/.venv/bin/python -m power_load_forecasting sync
StandardOutput=append:/var/log/power_load_sync.log
StandardError=append:/var/log/power_load_sync.log
```

## 6.3 timer 示例

每小时执行：

```ini
[Unit]
Description=Run power load forecasting sync every hour

[Timer]
OnCalendar=hourly
Persistent=true
Unit=power-load-sync.service

[Install]
WantedBy=timers.target
```

## 6.4 常用命令

启用：

```bash
systemctl daemon-reload
systemctl enable --now power-load-sync.timer
```

查看：

```bash
systemctl status power-load-sync.timer
systemctl list-timers --all | grep power-load-sync
```

手动测试：

```bash
systemctl start power-load-sync.service
systemctl status power-load-sync.service
```

看日志：

```bash
tail -n 100 /var/log/power_load_sync.log
journalctl -u power-load-sync.service -n 100 --no-pager
```

## 7. 并发与冲突建议

当前代码**不建议并发执行多个采集任务**。

会冲突的典型情况：

- `sync` 和 `backfill-actual` 同时运行
- `sync` 和 `backfill-forecast-snapshots` 同时运行
- `sync` 和 `backfill-forecast-window` 同时运行
- 两个 `sync` 同时运行
- 两个历史回补任务同时运行

原因：

- `collector_state.json` 是整体读写，存在覆盖风险
- parquet 分区写入存在并发覆盖风险
- forecast 去重是业务逻辑，不是锁机制

## 7.1 正确操作方式

### 日常运行

- 只保留一个 `sync` 定时器

### 需要回填时

先停定时任务：

```bash
systemctl stop power-load-sync.timer
systemctl stop power-load-sync.service
```

然后执行回填：

```bash
cd /opt/power_load_forecasting
python -m power_load_forecasting backfill-forecast-snapshots
python -m power_load_forecasting backfill-forecast-window --start-date 2026-03-01 --end-date 2026-04-20
python -m power_load_forecasting backfill-actual --start-date 2023-01-01
```

回填完再恢复：

```bash
systemctl start power-load-sync.timer
```

## 7.2 后续代码增强建议

建议后续加统一文件锁，例如：

- `/data/power_load_forecasting/meta/collector.lock`

这样可以从代码层面强制同一时刻只允许一个任务运行。

## 8. 数据存储建议

## 8.1 数据目录要持久化

必须保证以下目录不是临时盘：

- `data/raw`
- `data/curated`
- `data/meta`

推荐把数据根目录挂到独立盘：

```toml
data_root = "/data/power_load_forecasting"
```

## 8.2 定期检查磁盘

建议每周检查：

```bash
du -sh /data/power_load_forecasting
df -h
```

尤其 forecast snapshot 长期运行后增长会比较快。

## 8.3 备份建议

至少备份：

- `data/meta/collector_state.json`
- `config/collector.toml`

如果数据很关键，建议备份整个 `data/`。

## 9. 日志与监控建议

## 9.1 日志建议

日志文件建议：

```text
/var/log/power_load_sync.log
```

建议定期查看：

```bash
tail -n 200 /var/log/power_load_sync.log
```

## 9.2 建议关注的异常

重点关注：

- HTTP 429
- 连接超时
- JSON 解析错误
- parquet 写入失败
- `forecast_unchanged` 是否长期不变
- `actual_last_complete_date` 是否停滞

## 9.3 状态检查

定期运行：

```bash
cd /opt/power_load_forecasting
python -m power_load_forecasting status
```

重点看：

- `actual_last_complete_date`
- `forecast_last_issue_time_utc`
- `forecast_last_checked_at_utc`
- `forecast_snapshot_count`

如果 `forecast_last_checked_at_utc` 在更新，但 `forecast_snapshot_count` 长期不变，说明系统在正常检查，只是 forecast 没变化。

## 10. 更新代码建议

更新代码时建议这样做：

```bash
cd /opt/power_load_forecasting
systemctl stop power-load-sync.timer
systemctl stop power-load-sync.service

git pull
source .venv/bin/activate
pip install -r requirements.txt

python -m power_load_forecasting status
python -m power_load_forecasting sync

systemctl start power-load-sync.timer
```

不要在定时任务运行过程中直接 `git pull`。

## 11. 安全与权限建议

建议不要长期用 `root` 跑采集器，最好创建专用用户，例如：

```bash
useradd -r -s /bin/bash powerforecast
```

然后：

- 代码目录归这个用户
- 数据目录归这个用户
- systemd service 指定 `User=powerforecast`

这样更稳，也更符合生产规范。

## 12. 推荐的运维流程

## 首次上线

```bash
1. 拉代码
2. 建 venv
3. 安装依赖
4. 配置 collector.toml
5. 跑 status
6. 回填 forecast snapshots
7. 回填 actuals
8. 手动跑 sync
9. 配置并启用 timer
```

## 日常运行

```bash
1. systemd timer 定时跑 sync
2. 定期看日志
3. 定期看 status
4. 监控磁盘占用
```

## 需要补历史时

```bash
1. 停 timer
2. 停 service
3. 先跑低频全局 backfill
4. 有需要再跑高频窗口 backfill
4. 验证 status
5. 恢复 timer
```

## 14. 最简结论

生产上建议这样落地：

- 用 `systemd timer`
- 每小时执行一次 `sync`
- 历史回填默认采用“低频全局 + 高频窗口”的双模式
- 低频全局回填默认只覆盖 `2026-01-01` 之后的数据
- 高频窗口回填只用于少量关键日期区间
- 回填前暂停定时同步
- 数据目录放持久盘
- 定期检查 `status` 和日志

如果你需要，我下一步可以继续给你补两份文件：

- `systemd` 的完整 service/timer 模板
- `logrotate` 的日志轮转配置模板
